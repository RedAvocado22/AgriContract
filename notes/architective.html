📦 Architecture

Clean Architecture — Package Structure

Mỗi service được chia thành 3 layer. Dependency chỉ chạy từ ngoài vào trong — infrastructure phụ thuộc vào domain, không bao giờ ngược lại.

src/main/java/com/agricontract/{service}/
│
├── domain/              ← Lớp trong cùng — pure Java, không import Spring/JPA
│   ├── model/           ← Aggregate Roots, Entities
│   │   └── vo/          ← Value Objects (record)
│   ├── event/           ← Domain Events
│   └── repository/      ← Repository interfaces (ports) — chỉ là interface
│
├── application/         ← Use cases và DTOs
│   ├── usecase/         ← Interface cho từng use case
│   └── dto/             ← Command (input) và Response (output)
│
└── infrastructure/      ← Lớp ngoài — code Spring/JPA/RabbitMQ ở đây
    ├── persistence/
    │   ├── entity/      ← JPA @Entity — KHÁC với Domain model
    │   ├── mapper/      ← MapStruct: Domain ↔ JPA entity
    │   └── repository/  ← Implement domain repository interface
    ├── messaging/        ← RabbitMQ publisher/consumer
    ├── web/             ← @RestController
    └── config/          ← Spring @Configuration beans
✅ Domain layer

Pure Java. Không có annotation Spring nào. Không biết database hay HTTP tồn tại. Đây là nơi viết business logic.

✅ Application layer

Orchestrate use cases. Nhận Command từ controller, gọi domain model, lưu qua repository port.

✅ Infrastructure layer

Implement các interface từ domain. Chứa code Spring, JPA, RabbitMQ. Có thể swap mà không ảnh hưởng domain.

❌ Không được làm

Domain import @Entity, @Service của Spring. JPA entity lẫn lộn với domain model.

Tại sao JPA Entity tách với Domain Model?

Domain model Contract.java có business methods (sign(), settle()...), không có JPA annotation. JPA entity ContractJpaEntity.java chỉ là "table mapping" thuần túy. MapStruct convert giữa 2 thứ này.

// domain/model/Contract.java — pure business logic
@Getter
public class Contract {
    private ContractId contractId;
    private ContractStatus status;

    public void sign(String actorId) { /* business logic */ }
}

// infrastructure/persistence/entity/ContractJpaEntity.java — chỉ là table
@Entity @Table(name = "contracts")
public class ContractJpaEntity {
    @Id private Long id;
    @Column private String contractId;
    @Enumerated(EnumType.STRING) private ContractStatus status;
    // không có business method nào
}
🧩 DDD

Domain-Driven Design — Aggregate, VO, Entity

Value Object (VO) — Java Record

Immutable. Không có identity — so sánh bằng value, không phải reference. Dùng record của Java 16+.

// Ví dụ: UserId, Money, Quantity, ContactInfo
public record Money(BigDecimal amount, String currency) {}

// Tại sao không dùng String thẳng?
void pay(String amount, String currency)  // ← dễ nhầm thứ tự tham số
void pay(Money money)                    // ← rõ ràng, type-safe

Entity

Có identity (id). Có thể thay đổi state theo thời gian. Trong project này: EscrowTransaction là entity (có transactionId, append-only).

Aggregate Root

Là "cửa ngõ" vào một nhóm objects liên quan. Chỉ có thể modify thông qua Aggregate Root, không được sửa trực tiếp các object bên trong.

🏠 UserProfile — Aggregate Root

Owns: ContactInfo (VO embed), Role, VerificationStatus
Methods: create(), verify(), reject()

📦 Listing — Aggregate Root

Owns: Quantity, Money (VOs)
Methods: create(), close(), expire()
State: ACTIVE → CLOSED | EXPIRED

📄 Contract — Aggregate Root ⭐

Phức tạp nhất. 8 business methods. Accumulate domain events. Snapshot data từ Listing tại thời điểm offer.

💰 EscrowAccount — Aggregate Root

Owns: List<EscrowTransaction> (append-only ledger)
Methods: lock(), release(), penalize()

Aggregate Root không có public constructor. Tạo instance chỉ qua static factory để đảm bảo invariants luôn đúng từ lúc tạo.

Snapshot — Tại sao Contract lưu lại productName, quantity?

📬 Pattern

Outbox Pattern

Vấn đề cần giải quyết

❌
Nếu không dùng Outbox Contract-service save DB thành công nhưng publish RabbitMQ thất bại → escrow-service không bao giờ nhận được event → escrow không bao giờ được lock → hệ thống inconsistent.

Giải pháp

Ghi event vào bảng contract_domain_events trong cùng một DB transaction với việc update contract. Một scheduler riêng đọc và publish.

// Trong cùng 1 DB transaction:
UPDATE contracts SET status = 'SIGNED' WHERE contract_id = ?
INSERT INTO contract_domain_events (event_id, event_type, payload, status)
  VALUES (?, 'contract.signed', ?, 'PENDING')

// OutboxPoller (@Scheduled mỗi 1 giây):
SELECT * FROM contract_domain_events WHERE status = 'PENDING' LIMIT 50
  → publish to RabbitMQ
  → UPDATE status = 'PUBLISHED'
📋 Bảng contract_domain_events

event_id — UUID unique
event_type — vd: "contract.signed"
aggregate_id — contractId
payload — JSON của toàn bộ event
status — PENDING | PUBLISHED | FAILED
published_at — timestamp khi publish thành công

💡
Phase 1 vs Phase 2 Phase 1 dùng @Scheduled polling đơn giản. Phase 2 có thể nâng lên Debezium CDC (Change Data Capture) để giảm latency và tránh polling tốn resource.
🎭 Pattern

Choreography Saga

Saga là cách handle distributed transaction khi không dùng 2-phase commit. Project dùng Choreography — không có orchestrator trung tâm.

Choreography vs Orchestration

✅ Choreography (đang dùng)

Mỗi service tự biết phải làm gì khi nhận event. Không có "god service" điều phối. Services độc lập hơn.

❌ Orchestration (không dùng)

Có 1 Saga Orchestrator service điều phối toàn bộ flow. Phức tạp hơn, tạo single point of failure.

Happy path flow

contract-service  → publish contract.signed
  escrow-service   → subscribe → lock escrow
  notif-service    → subscribe → send email

contract-service  → publish contract.settled
  escrow-service   → subscribe → release escrow
  notif-service    → subscribe → send email

Compensating transactions (khi fail)

Choreography Saga không tự rollback. Mỗi service phải implement "compensating action" khi nhận cancel event.

// Nếu buyer cancel sau khi OFFERED (escrow chưa lock):
contract-service  → publish contract.cancelled
  notif-service    → subscribe → notify both parties
  // escrow-service không cần làm gì — chưa lock

// Nếu dispute → arbitrate (escrow đang LOCKED):
contract-service  → publish contract.arbitrated
  escrow-service   → subscribe → penalize buyer/seller
🔑 Pattern

Idempotency

RabbitMQ dùng at-least-once delivery — một event có thể được deliver nhiều lần. Mỗi consumer phải đảm bảo xử lý cùng event 2 lần không gây ra side effect.

Escrow Service — contractId là idempotency key

public void onContractSigned(Map event) {
    String contractId = event.get("contractId");

    // Check trước khi xử lý
    if (escrowRepo.existsByContractId(contractId)) {
        log.warn("Escrow already locked for contract {}, skipping", contractId);
        return;  // bỏ qua, không lock lần 2
    }

    EscrowAccount.lock(contractId, buyerId, sellerId, amount);
}

Notification Service — eventId là dedup key

public void onContractSigned(Map event) {
    String eventId = event.get("eventId");

    if (notifRepo.existsByEventId(eventId)) {
        return;  // email đã gửi rồi, skip
    }

    // Gửi email + lưu log
    sendEmail(...);
    notifRepo.save(NotificationLog.of(eventId, ...));
}

🔐 Security

Keycloak & JWT Authentication

Flow xác thực

1. Frontend → Keycloak :8080 (login với username/password)
   ← JWT access token

2. Frontend → API Gateway :8888
   Header: Authorization: Bearer <jwt>

3. API Gateway → Keycloak JWKS endpoint (validate chữ ký JWT)
   → inject header X-User-Id: <keycloak sub>
   → forward request đến service

4. Downstream services — tin header X-User-Id
   Không cần gọi lại Keycloak (tốn network)
   Chỉ an toàn vì request đến từ Gateway qua internal Docker network

JWT Claims quan trọng

ClaimÝ nghĩaDùng để
subKeycloak user UUIDuserId trong toàn bộ hệ thống
emailEmail người dùngGửi notification
realm_access.rolesSELLER / BUYER / ADMINAuthorization tại Gateway + service
⚠️
Phase 1 limitation Downstream services tin header X-User-Id mà không verify. Điều này an toàn vì request chỉ đến từ Gateway qua Docker internal network. Phase 2 sẽ thêm mTLS hoặc signed header để tăng bảo mật.

user-service vs Keycloak

Keycloak giữ credentials (password, token). user-service giữ profile (tên công ty, role, địa chỉ). Field userId trong user_profiles = Keycloak sub. Không bao giờ duplicate password vào user-service.

⚙️ Services

Chi tiết từng Service

contract-service — Service quan trọng nhất

Implement Contract aggregate với đầy đủ 8 business methods:

public class Contract {
    // Factory
    public static Contract offer(...)           // OFFERED

    // Happy path
    public void sign(String actorId)            // → SIGNED
    public void confirmDelivery(String actorId) // → GOODS_DELIVERED
    public void settle(String actorId)          // → SETTLED

    // Cancel (chỉ khi OFFERED)
    public void cancelByBuyer(String actorId, String reason)
    public void cancelBySeller(String actorId, String reason)

    // Dispute (khi SIGNED)
    public void dispute(String actorId, String reason)
    public void arbitrate(Money penalty, boolean penalizeBuyer)

    // Outbox — drain events để persist vào outbox table
    public List<DomainEvent> pullDomainEvents()
}

product-service — Sync call quan trọng

Khi contract-service nhận request sign(), nó phải gọi sync đến product-service để close listing trước khi emit event. Tại sao?

💡
Tránh race condition Nếu close listing là async, có thể xảy ra race: 2 buyer cùng lúc sign 2 contract cho cùng 1 listing trước khi listing kịp close. Sync call đảm bảo listing CLOSED trước khi contract.signed được emit.

escrow-service — Append-only ledger

EscrowTransaction là append-only — không bao giờ UPDATE, chỉ INSERT. Mỗi operation tạo 1 record mới. Đây là audit trail tài chính.

// Mỗi lần lock/release/penalty → thêm 1 row vào escrow_transactions
// KHÔNG update row cũ
escrow_transactions:
  LOCK    | 50,000,000 VND | 2026-06-01 ...
  RELEASE | 50,000,000 VND | 2026-06-15 ...  ← row mới

notification-service — Retry với exponential backoff

// Retry 3 lần trước khi mark FAILED
attempt 1: send email → fail → wait 1s
attempt 2: send email → fail → wait 2s
attempt 3: send email → fail → wait 4s
→ status = FAILED, cần manual intervention
