Luồng Request Tổng Quát

Mỗi request từ frontend đều đi qua đúng một cổng duy nhất: API Gateway. Từ đó mới phân phối xuống các service.

Frontend
→
API Gateway :8888
→
Service :808x
→
MySQL

Từng bước chi tiết

1
Frontend gửi request kèm JWT

Header: Authorization: Bearer <jwt>. Frontend không cần biết service nào đang phục vụ.

2
Gateway validate JWT với Keycloak

Gateway gọi Keycloak JWKS endpoint để verify chữ ký token. Nếu token hết hạn hoặc sai chữ ký → trả 401 ngay, không cho vào trong.

3
Gateway inject header X-User-Id

Sau khi JWT hợp lệ, Gateway đọc claim sub (Keycloak UUID) và gắn vào header X-User-Id trước khi forward xuống service.

4
Service xử lý request

Service nhận request, đọc X-User-Id từ header. Không cần gọi lại Keycloak — tin tưởng Gateway đã validate.

Tại sao thiết kế như vậy?

Nếu mỗi service tự validate JWT, mỗi service phải gọi Keycloak → tốn network, tốn CPU, và mỗi khi Keycloak đổi public key thì phải cập nhật tất cả service. Gateway làm một lần duy nhất, các service downstream không cần quan tâm đến auth — họ chỉ cần xử lý business logic.

⚠️
An toàn vì mạng nội bộ Service tin X-User-Id mà không verify — điều này chỉ an toàn vì các service chỉ nhận request từ Gateway qua Docker internal network. Không có service nào expose port ra ngoài ngoại trừ Gateway.
🔐 Auth

Luồng Xác Thực — Keycloak & JWT

Hệ thống tách biệt rõ ràng: Keycloak giữ credentials (password, token), user-service giữ profile (tên công ty, địa chỉ, role business).

Bước 1 — Đăng nhập lấy token

Frontend
Gửi POST /realms/agricontract/protocol/openid-connect/token với username + password đến Keycloak :8080
Keycloak
Verify credentials → trả về access_token (JWT, 15 phút) + refresh_token (30 ngày)

Bước 2 — Gọi API với token

Frontend
Gửi request đến Gateway :8888 kèm Authorization: Bearer <jwt>
Gateway
Lấy public key từ Keycloak JWKS (/.well-known/jwks.json), verify chữ ký JWT
Gateway
Đọc sub → thêm header X-User-Id: <uuid>. Đọc realm_access.roles → check quyền. Forward request xuống service.

Cấu trúc JWT

{
  "sub": "a3f2c1d4-...",          // userId dùng trong toàn bộ hệ thống
  "email": "seller@htx.vn",
  "realm_access": {
    "roles": ["SELLER"]            // SELLER | BUYER | ADMIN
  },
  "exp": 1748900000,              // hết hạn sau 15 phút
  "iss": "http://keycloak/realms/agricontract"
}
Tại sao tách Keycloak với user-service?

Keycloak là "chuyên gia" về identity — nó handle OAuth2, token rotation, brute-force protection, MFA. user-service tập trung vào business profile. Nếu gộp lại, user-service phải implement toàn bộ security infrastructure từ đầu — vừa tốn công vừa dễ sai. Dùng Keycloak = đứng trên vai giants.

✅ Hợp đồng

Happy Path — Hợp Đồng Thành Công

Luồng lý tưởng: buyer offer → seller ký → seller giao hàng → buyer xác nhận → tiền được giải phóng.

OFFERED
→
SIGNED
→
GOODS_DELIVERED
→
SETTLED

Bước 1 — Buyer tạo offer

Buyer
POST /contracts — chọn listing, điền số lượng, giá
contract-svc
Contract.offer() — snapshot lại productName, quantity, agreedPrice từ listing ngay lúc này. Status → OFFERED
Tại sao snapshot dữ liệu listing?

Sau khi hợp đồng được ký, seller có thể sửa listing (đổi giá, đổi số lượng). Nếu contract chỉ lưu listingId và đọc live, điều khoản hợp đồng sẽ bị thay đổi âm thầm. Snapshot đảm bảo hợp đồng luôn phản ánh đúng thỏa thuận tại thời điểm offer.

Bước 2 — Seller ký hợp đồng

Seller
PUT /contracts/{id}/sign
contract-svc
Gọi sync product-service → CLOSE listing (tránh race condition)
contract-svc
Contract.sign() → Status: SIGNED. Lưu ContractSignedEvent vào outbox table trong cùng 1 transaction
escrow-svc
Nhận contract.signed event → EscrowAccount.lock() — khóa tiền của buyer
Tại sao close listing phải là sync call?

Nếu close listing là async (qua event), có thể xảy ra race condition: buyer A và buyer B cùng lúc sign 2 contract cho cùng 1 listing trước khi listing kịp close. Sync call đảm bảo listing CLOSED ngay lập tức, trước khi response trả về buyer. Seller chỉ bán được cho 1 người.

Bước 3 — Seller xác nhận giao hàng

Seller
PUT /contracts/{id}/deliver
contract-svc
Status → GOODS_DELIVERED. Emit GoodsDeliveredEvent
notif-svc
Nhận event → gửi email cho buyer: "Hàng đã giao, vui lòng xác nhận để giải phóng thanh toán"

Bước 4 — Buyer xác nhận nhận hàng

Buyer
PUT /contracts/{id}/settle
contract-svc
Status → SETTLED. Emit ContractSettledEvent
escrow-svc
Nhận event → EscrowAccount.release() — chuyển tiền cho seller
notif-svc
Gửi email xác nhận thanh toán cho cả buyer lẫn seller
Tại sao escrow cần chờ buyer confirm?

Nếu tiền tự động chuyển ngay khi seller nói "đã giao", seller có thể gian lận: nhập delivery mà không giao thật. Buyer xác nhận là bằng chứng nhận hàng. Escrow giữ tiền làm đòn bẩy — seller có động lực giao hàng thật, buyer có bảo đảm tiền không mất nếu hàng chưa đến.

❌ Cancel

Cancel Flow — Hủy Hợp Đồng

Cancel chỉ được phép khi contract đang ở trạng thái OFFERED. Sau khi đã SIGNED thì chỉ có thể dispute.

❌ Nếu cancel sau SIGNED

Escrow đã khóa tiền của buyer. Cần compensating transaction để mở khóa. Phức tạp hơn nhiều, và ai trả phí giao dịch?

✅ Chỉ cancel khi OFFERED

Escrow chưa lock — không cần rollback gì cả. Đơn giản và sạch. Buyer đổi ý trước khi seller đồng ý thì không ai thiệt.

Buyer hủy (rút offer)

Buyer
PUT /contracts/{id}/cancel
contract-svc
Guard: chỉ cho phép nếu status == OFFERED và actorId == buyerId. Contract.cancelByBuyer() → Status: CANCELLED_BY_BUYER
notif-svc
Notify seller: "Buyer đã rút offer"

Seller từ chối

Seller
PUT /contracts/{id}/reject
contract-svc
Guard: status == OFFERED và actorId == sellerId. Contract.cancelBySeller() → Status: CANCELLED_BY_SELLER
notif-svc
Notify buyer: "Seller đã từ chối offer của bạn"
Tại sao phân biệt CANCELLED_BY_BUYER và CANCELLED_BY_SELLER?

Để có audit trail rõ ràng — ai là người hủy? Khi có tranh chấp sau này (ngoài app), hệ thống có thể chứng minh ai rút lui trước. Cũng giúp analytics: seller từ chối nhiều = có vấn đề về chất lượng listing; buyer rút nhiều = có vấn đề về UX hoặc giá.

⚖️ Dispute

Dispute Flow — Tranh Chấp

Khi hợp đồng đã SIGNED và buyer không nhận được hàng hoặc hàng sai, buyer có thể mở tranh chấp thay vì bị ép settle.

SIGNED
→
DISPUTED
→
ARBITRATED

Bước 1 — Buyer mở tranh chấp

Buyer
PUT /contracts/{id}/dispute kèm lý do
contract-svc
Guard: status == SIGNED. Contract.dispute() → Status: DISPUTED
escrow-svc
Escrow vẫn LOCKED — tiền chưa đi đâu, chờ kết quả phán xử
notif-svc
Notify admin: "Có tranh chấp mới cần xử lý"

Bước 2 — Admin phán xử

Admin
PUT /contracts/{id}/arbitrate kèm penalizeBuyer: true/false và số tiền penalty
contract-svc
Contract.arbitrate() → Status: ARBITRATED. Emit ContractArbitratedEvent
escrow-svc
Nhận event → EscrowAccount.penalize():
— penalizeBuyer=false: seller sai → hoàn tiền cho buyer (PENALIZED_SELLER)
— penalizeBuyer=true: buyer sai → chuyển tiền cho seller (PENALIZED_BUYER)
Tại sao cần DISPUTED state thay vì cancel thẳng?

Cancel thẳng khi SIGNED sẽ không xử lý được vấn đề "ai sai?". DISPUTED state là "phòng chờ" — trong khi admin điều tra, tiền vẫn bị giữ (không ai rút được), hệ thống có thể lưu evidence, admin có thể xem lịch sử. Nó cũng ngăn seller tự release escrow trong lúc đang tranh chấp.

📬 Pattern

Outbox Pattern — Tại Sao Cần?

Vấn đề gốc rễ

❌ Không có Outbox

Save contract DB thành công → publish RabbitMQ thất bại (network blip) → escrow không bao giờ nhận được event → hệ thống inconsistent mãi mãi.

✅ Với Outbox

Event được ghi vào DB trong cùng transaction với contract update. Dù RabbitMQ chết, khi nó sống lại, poller sẽ publish lại. Không bao giờ mất event.

Luồng Outbox

1
DB Transaction — atomic

Trong cùng 1 transaction: UPDATE contracts SET status='SIGNED' + INSERT vào contract_domain_events (status=PENDING). Cả hai thành công hoặc cả hai rollback. Không có trạng thái nửa vời.

2
OutboxPoller chạy mỗi 1 giây

@Scheduled query: SELECT * FROM contract_domain_events WHERE status='PENDING' LIMIT 50. Chạy độc lập với request, không block business flow.

3
Publish lên RabbitMQ

Mỗi event được publish với routing key = event_type. Nếu RabbitMQ down → retry ở lần poll sau. Event không bị mất.

4
Mark PUBLISHED

Sau khi publish thành công → UPDATE status='PUBLISHED', ghi published_at. Audit trail đầy đủ.

// 1 DB transaction duy nhất — atomicity đảm bảo hai việc cùng thành công/fail
UPDATE contracts
  SET status = 'SIGNED', updated_at = NOW()
  WHERE contract_id = ?

INSERT INTO contract_domain_events
  (event_id, event_type, aggregate_id, payload, status, created_at)
  VALUES (?, 'contract.signed', ?, ?, 'PENDING', NOW())
Tại sao không dùng @TransactionalEventListener?

Spring's @TransactionalEventListener publish sau khi transaction commit — nếu publish fail, không có cơ chế retry. Outbox pattern thì poller retry đến khi thành công. Đây là sự khác biệt giữa "best-effort" và "at-least-once delivery".

💡
Phase 2 nâng cấp lên Debezium CDC Thay vì polling DB mỗi giây (tốn I/O), Debezium đọc MySQL binary log như một replica. Latency gần như 0, không tốn query. Nhưng phức tạp hơn để setup nên Phase 1 dùng polling trước.
🎭 Pattern

Choreography Saga — Distributed Transaction

Không có distributed transaction (2PC) trong microservices. Saga là cách handle "chuỗi hành động cần nhất quán" qua nhiều service.

Choreography vs Orchestration

✅ Choreography (đang dùng)

Mỗi service tự biết phải làm gì khi nhận event. Không có "god service" ở trung tâm. Services độc lập, có thể deploy riêng.

❌ Orchestration (không dùng)

Cần 1 Saga Orchestrator service điều phối. Orchestrator trở thành single point of failure và bottleneck. Thêm 1 service phải maintain.

Happy Path Saga

contract-svc
Publish contract.signed lên RabbitMQ exchange agricontract.events
escrow-svc
Subscribe contract.signed → lock escrow. Độc lập, không biết notif-svc tồn tại.
notif-svc
Subscribe contract.signed → gửi email. Cùng event, 2 consumer khác nhau xử lý song song.
Tại sao cùng 1 event có 2 consumer?

RabbitMQ Topic Exchange: mỗi service tạo queue riêng bind vào cùng routing key. escrow-svc có queue escrow.contract.signed, notif-svc có queue notif.contract.signed. Cả hai đều nhận bản copy của event, xử lý song song, không block nhau. Contract-svc không cần biết ai đang lắng nghe.

Compensating Transaction — Khi có lỗi

Choreography Saga không tự rollback. Mỗi service implement "compensating action" khi nhận cancel/arbitrate event.

contract-svc
Publish contract.arbitrated kèm penalizeBuyer: false
escrow-svc
Compensate: unlock escrow + hoàn tiền buyer (PENALIZED_SELLER). Đây là "undo" của bước lock.
notif-svc
Gửi email thông báo kết quả phán xử cho cả hai bên.
⚠️
Saga không đảm bảo isolation Trong khoảng thời gian giữa các bước, hệ thống có thể ở trạng thái "partially committed" — contract đã SIGNED nhưng escrow chưa kịp lock. Đây là đánh đổi của microservices. Trong project này window đó rất nhỏ (dưới 1 giây với outbox poller).
💰 Escrow

Escrow Flow — Quản Lý Tiền

Escrow là "két sắt trung gian" — tiền buyer được khóa lại khi ký hợp đồng, chỉ được giải phóng khi có xác nhận rõ ràng.

Lifecycle của EscrowAccount

1
LOCK — khi contract SIGNED

Nhận contract.signed → EscrowAccount.lock(contractId, buyerId, sellerId, amount). Tạo record LOCK trong escrow_transactions. Status escrow: LOCKED.

2
Chờ — tiền bất động trong escrow

Không ai có thể rút trong giai đoạn này. Seller có động lực giao hàng. Buyer có bảo đảm.

3a
RELEASE — khi contract SETTLED

Nhận contract.settled → EscrowAccount.release(). Thêm row RELEASE vào ledger. Tiền đến tay seller.

3b
PENALIZE — khi ARBITRATED

Nhận contract.arbitrated → EscrowAccount.penalize(penalty, penalizeBuyer). Thêm row PENALIZED_BUYER hoặc PENALIZED_SELLER.

Append-only Ledger

-- Mỗi operation là 1 row mới, KHÔNG UPDATE row cũ
escrow_transactions:
  id | contract_id | type    | amount      | created_at
  1  | abc-123     | LOCK    | 50,000,000  | 2026-06-01 09:00
  2  | abc-123     | RELEASE | 50,000,000  | 2026-06-15 14:30  ← row mới

-- KHÔNG làm thế này:
UPDATE escrow_accounts SET status = 'RELEASED' WHERE contract_id = 'abc-123'
Tại sao append-only? Không update status thôi?

Append-only là audit trail tài chính không thể sửa. Nếu update status thì mất lịch sử: tiền bị lock lúc mấy giờ? Ai release? Có penalty không? Với ledger, mọi sự kiện đều có timestamp và không thể xóa. Đây là tiêu chuẩn của bất kỳ hệ thống tài chính nào.

✅
Idempotency tự nhiên Vì escrow_accounts.contract_id có UNIQUE constraint, nếu contract.signed được deliver 2 lần (at-least-once), lần 2 sẽ bị reject bởi DB constraint. Không cần logic check thủ công.
🔑 Pattern

Idempotency — Xử Lý Duplicate Event

RabbitMQ đảm bảo at-least-once delivery — nghĩa là một event có thể đến 2 lần hoặc nhiều hơn (sau khi retry, sau khi restart). Consumer phải xử lý an toàn.

Tại sao event bị duplicate?

1
Consumer xử lý xong nhưng chưa kịp ACK

Consumer crash sau khi gửi email nhưng trước khi gửi ACK. RabbitMQ nghĩ chưa xử lý → redeliver. Email được gửi 2 lần.

2
Outbox poller publish 2 lần

Poller publish thành công nhưng crash trước khi UPDATE status='PUBLISHED'. Lần poll sau publish lại.

Giải pháp từng service

💰 escrow-service — dùng contractId

UNIQUE constraint trên escrow_accounts.contract_id. Nếu lock lần 2 → INSERT bị reject → bắt exception → log warn và skip. Không có code check thủ công.

public void onContractSigned(ContractSignedEvent event) {
    // Không cần if-check — DB tự throw DataIntegrityViolationException
    try {
        escrowRepo.save(EscrowAccount.lock(event.contractId(), ...));
    } catch (DataIntegrityViolationException e) {
        log.warn("Duplicate event for contract {}, skipping", event.contractId());
    }
}
📧 notification-service — dùng eventId

UNIQUE constraint trên notification_logs.event_id. Mỗi event có UUID riêng — lần 2 bị reject. Email không gửi 2 lần.

public void onContractSigned(ContractSignedEvent event) {
    if (notifRepo.existsByEventId(event.eventId())) {
        return;  // đã xử lý rồi, bỏ qua
    }
    sendEmail(event);
    notifRepo.save(NotificationLog.of(event.eventId(), ...));
}
Tại sao escrow dùng contractId còn notification dùng eventId?

Với escrow, 1 contract chỉ được lock 1 lần — contractId là natural idempotency key. Với notification, cùng 1 contract có nhiều event (signed, delivered, settled...) mỗi cái cần gửi email riêng — nên phải dùng eventId (unique per event type per contract). Dùng contractId cho notification sẽ chặn nhầm cả email hợp lệ.

ServiceKeyCơ chếLý do
escrow-service contractId DB UNIQUE constraint 1 contract = 1 escrow account
notification-service eventId existsByEventId check 1 contract có nhiều event khác nhau
outbox poller event_id UNIQUE + status filter Chỉ publish row PENDING, sau đó mark PUBLISHED
